# Rust Interop การเดินทางสู่แก่นแท้ของ Safety บนตะเข็บรอยต่อ C/C++

> 📅 วันที่เผยแพร่: 2026-01-30

โค้ดเบสเดิมที่เขียนด้วย C และ C++ การทำงานร่วมกัน (Interoperability หรือ Interop) คือด่านสำคัญที่หลีกเลี่ยงไม่ได้ การข้ามตะเข็บรอยต่อระหว่างภาษาเหล่านี้คือจุดที่ความปลอดภัยของ #Rust ถูกท้าทายอย่างแท้จริง เพราะ FFI (Foreign Function Interface) ไม่ได้เป็นเพียงการเรียกฟังก์ชันข้ามภาษา แต่คือการจัดการกับ Undefined Behavior ที่ซ่อนเร้นอยู่

ความท้าทายนี้ได้รับการสำรวจอย่างลึกซึ้งในงานวิจัยเชิงวิชาการ เรื่อง "Fortifying the Seams Between C/C++ and Rust: Characterizing Bugs in Interop Tools" ซึ่งไม่ได้มุ่งเน้นไปที่ปัญหาพื้นฐานอย่าง Tool Crash หรือ Performance แต่เจาะลึกไปที่ปัญหาที่ซับซ้อนกว่า นั่นคือความไม่สามารถสร้างโค้ดที่ถูกต้องและใช้งานได้จริง ซึ่งหมายถึงโค้ดที่สร้างขึ้นนั้น ไม่ซื่อสัตย์ต่อความตั้งใจเดิม (Unfaithful to the original intent) ของโค้ดต้นฉบับ การทำความเข้าใจข้อบกพร่องเหล่านี้จึงเป็นกุญแจสำคัญสำหรับ Developer ในการสร้าง Abstraction ที่ปลอดภัยเหนือโค้ด `unsafe`

การเดินทางของ Interop ใน #Rust ถูกขับเคลื่อนด้วยเครื่องมือหลักสามตัว ซึ่งแต่ละตัวมีปรัชญาในการจัดการ "รอยต่อ" ที่แตกต่างกัน BINDGEN เป็นเครื่องมือแบบ Translation-based ที่ได้รับความนิยมสูงสุด โดยใช้ `libclang` เพื่อแปล C Header ให้เป็น Rust Binding โดยอัตโนมัติ ในทางกลับกัน CBINDGEN ทำหน้าที่สร้าง C Header จากโค้ด #Rust เพื่อให้โค้ด #Rust สามารถถูกเรียกใช้จาก C/C++ ได้ และสุดท้ายคือ CXX ซึ่งใช้แนวทางแบบ Bridge-based ที่พยายามสร้าง Type-safe Bridge โดยใช้ Rust Types ที่ปลอดภัยเพื่อลดความเสี่ยงในการข้ามพรมแดนภาษา

แม้ว่าเครื่องมือเหล่านี้จะช่วยลดภาระในการเขียน Binding ด้วยมือ แต่ Developer ต้องตระหนักว่าความสะดวกสบายนี้มาพร้อมกับความเสี่ยงที่ซับซ้อน โดยเฉพาะอย่างยิ่งในเครื่องมือ Translation-based อย่าง BINDGEN ที่ต้องเผชิญกับความซับซ้อนของ C/C++ Type System

งานวิจัยได้จำแนกประเภทของข้อบกพร่องที่เกิดขึ้นที่ตะเข็บรอยต่อ ซึ่งเป็นความรู้ที่จำเป็นสำหรับผู้ที่ทำงานในระดับ Systems Programming

## 1. Memory Layout Mismatch (MEM)

ปัญหาที่ร้ายแรงที่สุดและเป็นสาเหตุหลักใน BINDGEN คือ Memory Layout Mismatch (MEM) ซึ่งคิดเป็น 17.3% ของข้อบกพร่องทั้งหมดที่รายงาน ปัญหานี้เกิดขึ้นเมื่อ Rust Binding ที่ถูกสร้างขึ้นมีขนาด (Size), การจัดเรียง (Alignment), หรือตำแหน่งของฟิลด์ (Offset) ในหน่วยความจำ ไม่ตรงกับที่คอมไพเลอร์ C/C++ กำหนดไว้ การไม่ตรงกันนี้มักเกิดจากการที่เครื่องมือ Interop ขาดความเข้าใจเชิงลึกในเรื่อง Data Layout ที่ขึ้นอยู่กับสถาปัตยกรรมและคอมไพเลอร์

เช่น เมื่อ Struct ใน C ถูกกำหนดให้เป็น `__packed__` หรือมีโครงสร้างที่ซับซ้อนอย่าง Bitfields และ Flexible Array Members (FAM) หาก BINDGEN แปล Attribute เหล่านี้ผิดพลาด Rust Struct ที่ได้จะมีการจัดเรียงหน่วยความจำที่ผิดไปจากเดิม เมื่อข้อมูลถูกส่งผ่านข้ามภาษา จะเกิดการอ่านหรือเขียนข้อมูลผิดตำแหน่ง ซึ่งนำไปสู่ Data Corruption หรือ Undefined Behavior ได้อย่างง่ายดาย

## 2. Unfaithful Code (UFC) และ Incomplete Code (INC)

นอกเหนือจากปัญหาเชิงกายภาพของหน่วยความจำแล้ว ยังมีปัญหาเชิงความหมาย (Semantic) ที่ท้าทายไม่แพ้กัน Unfaithful Code (UFC) คือกรณีที่โค้ดที่สร้างขึ้นนั้นคอมไพล์ผ่าน แต่ไม่รักษาความตั้งใจเดิมของโค้ดต้นฉบับ เช่น การกำหนด `mutability` ที่ผิดพลาด, การแมป Namespace ที่ไม่ถูกต้อง, หรือการละเลย `const` Qualifier ในขณะที่ Incomplete Code (INC) คือการขาดหายไปขององค์ประกอบที่จำเป็น เช่น การขาด Definition, Generic Arguments, หรือ Lifetime Bounds ซึ่งมักจะนำไปสู่ข้อผิดพลาดในการคอมไพล์หรือข้อผิดพลาดเชิงฟังก์ชันในภายหลัง

ข้อบกพร่องเหล่านี้ตอกย้ำว่าการทำ Interop ที่ปลอดภัยไม่ได้จบลงแค่การทำให้โค้ดคอมไพล์ได้ แต่ต้องมั่นใจว่าความหมายของโค้ดต้นฉบับยังคงอยู่ครบถ้วน

## 3. Triggers — ความซับซ้อนของ Type System

งานวิจัยระบุว่า Complex Structs เป็น Trigger ที่พบบ่อยที่สุดในทั้ง BINDGEN (30.8%) และ CBINDGEN (22%) ซึ่งชี้ให้เห็นว่าความซับซ้อนของ Type System ใน C/C++ เป็นอุปสรรคสำคัญที่สุดในการสร้าง Interop ที่เชื่อถือได้ นอกจากนี้ Tool Configuration ก็เป็นแหล่งที่มาของปัญหาที่สำคัญ การใช้ `blocklist` หรือ `allowlist` ใน BINDGEN ที่ไม่ถูกต้องอาจนำไปสู่การละเลย Dependency ที่จำเป็น ทำให้เกิดปัญหา Incomplete Code ได้ การจัดการกับ Primitive Data Types ก็ยังคงเป็นปัญหาที่น่าประหลาดใจ เช่น การแมป `size_t` หรือความแตกต่างของ `char` และ `wchar_t` บน Windows

สิ่งที่เราควรเรียนรู้จากงานวิจัยนี้คือ ความสามารถที่แท้จริงของ #Rust ในบริบท Interop คือการจำกัดขอบเขตของ `unsafe` ให้อยู่ในพื้นที่ที่เล็กที่สุดและควบคุมได้มากที่สุด การใช้ `unsafe` ใน FFI ไม่ได้เป็นเพียงการยอมรับความเสี่ยงของ Raw Pointers เท่านั้น แต่คือการรับผิดชอบต่อ Semantic Mismatches และ Layout Mismatches ที่เครื่องมืออัตโนมัติอาจมองข้าม

หากต้องทำงานกับ Struct ที่ซับซ้อน, Bitfields, หรือ Packed Structs การตรวจสอบ Memory Layout ด้วยตนเอง หรือการใช้แนวทาง Bridge-based อย่าง CXX ที่ให้ความปลอดภัยเชิง Type ที่สูงกว่า ควรถูกพิจารณาเป็นอันดับแรก CXX บังคับให้เราต้องทำการแมปที่ชัดเจน ซึ่งเป็นการแลกความสะดวกสบายกับการได้มาซึ่งความปลอดภัยที่เชื่อถือได้มากกว่า การรู้ว่าเครื่องมือ Interop ทำงานผิดพลาดที่จุดใด จะช่วยให้เราสามารถเขียน Safe Abstraction ครอบโค้ด `unsafe` ที่สร้างโดยเครื่องมือได้อย่างมีประสิทธิภาพ การทำ Interop ที่ดีจึงไม่ใช่แค่การทำให้โค้ดคอมไพล์ผ่าน แต่คือการสร้างรอยต่อที่แข็งแกร่งที่รักษาคุณสมบัติเด่นของ Rust ไว้ได้อย่างสมบูรณ์
