# มาลองสร้าง Semantic Version Control ด้วย Rust กัน

> 📅 วันที่เผยแพร่: 2026-04-21

มีเรื่องตลกจะเล่าให้ฟัง เวลาเราส่ง Pull Request หรือคุยกับเพื่อนในทีม เรามักจะพูดว่า เห้ย เพิ่งรีแฟกเตอร์ฟังก์ชันยืนยันตัวตนไปนะ เราแทบจะไม่พูดว่า นี่ๆ ฉันแก้บรรทัดที่ 47 ถึง 52 นะ 555 เพราะสมองมนุษย์ไม่ได้มองโค้ดเป็นบรรทัดไงครับ แต่น่าแปลกที่ระบบอย่าง Git ซึ่งเราใช้กันทุกวัน กลับมองโค้ดเป็นแค่บรรทัดข้อความ (Lines of text) มันไม่รู้จักเลยว่าอะไรคือ Function, Class หรือ Method ผลลัพธ์ที่ตามมาคือเรามักจะเจอกับ Merge conflict ปลอมๆ เพียงเพราะมีคนจัดฟอร์แมตโค้ดใหม่ ย้ายบรรทัด หรือแม้แต่ทำให้เครื่องมือ Code Review ที่พึ่งพา AI ทำงานได้แย่ลงเพราะขาดบริบท และนี่ไม่ใช่บั๊กของ Git แต่มันคือข้อจำกัดของ Abstraction ที่มันเลือกใช้ ซึ่งโปรเจกต์ sem และ weave ได้พยายามทลายข้อจำกัดนี้ด้วยการสร้าง Semantic Version Control ขึ้นมาใหม่ทั้งหมดโดยใช้ขุมพลังของภาษา #Rust เรามาดูกันครับ ว่าเขาทำยังไง

การจะแก้ปัญหานี้ได้ คำถามแรกที่ต้องตอบให้ได้ก่อนคือ อะไรคือหน่วยย่อยที่สุด (Atom) ที่เราจะนำมาประมวลผล? ถ้ามองเป็นระดับไฟล์มันดูจะก็หยาบเกินไป ไฟล์ที่มีการแก้แค่พิมพ์ผิดเทียบกับไฟล์ที่ถูกเขียนอัลกอริทึมใหม่แทบจะดูไม่ออกเลย ถ้ามองเป็นบรรทัดก็ละเอียดไปจนขาดความหมาย หรือถ้าเจาะลึกไประดับ AST Node เลยก็มีจำนวนมหาศาลและใกล้เคียงระดับ Compiler เกินกว่าจะนำมาใช้ประโยชน์ จุดลงตัวจึงมาตกอยู่ที่สิ่งที่เรียกว่า Entity ซึ่งก็คือพวก Function, Class, หรือ Method โครงสร้างข้อมูลหลักจึงถูกออกแบบผ่าน `SemanticEntity` struct ที่เก็บข้อมูลสำคัญทั้ง ID, ชื่อ, ประเภท, dependencies และที่สำคัญที่สุดคือ `parent_id` การมีฟิลด์นี้ทำให้ระบบสามารถจำลองลำดับชั้นได้ เช่น เมื่อระบบเจอ Class มันจะสกัด Method ย่อยๆ ออกมาเป็น Entity แยกต่างหากแล้วผูกกลับไปที่ Class แม่ ซึ่งโครงสร้างแบบนี้แหละที่เป็นไม้ตายสำคัญที่ทำให้เราสามารถทำ Merge โค้ดในระดับที่ลึกกว่าเดิมได้

เมื่อกำหนดโครงสร้างได้แล้ว ขั้นต่อไปคือการอ่านโค้ด โปรเจกต์นี้เลือกใช้ `tree-sitter` ในการแปลงโค้ดเป็น Concrete Syntax Tree เพราะมันเร็ว ทนทานต่อโค้ดที่พังหรือไม่สมบูรณ์ และรองรับภาษาแทบทั้งหมดบนโลก แต่ความท้าทายคือแต่ละภาษาก็เรียกชื่อ Node ต่างกันไป เช่น เวลาประกาศฟังก์ชัน Rust จะใช้ `function_item`, Python ใช้ `function_definition` ส่วน Go ใช้ `function_declaration` ผู้สร้างใช้วิธีออกแบบ Parser Plugin ให้เป็นแค่ Wrapper บางๆ แล้วใช้พลังของ `Trait` ในการจับคู่ (Mapping) Node เฉพาะเจาะจงของแต่ละภาษาให้กลายเป็น Entity ประเภทเดียวกัน การออกแบบสไตล์ Language-agnostic แบบนี้ทำให้ Pipeline การทำงานส่วนที่เหลือไม่ต้องสนใจเลยว่ากำลังจัดการกับภาษาอะไรอยู่ ส่งผลให้โปรเจกต์สามารถสเกลจากการรองรับ 4 ภาษา เป็น 26 ภาษาได้ภายในเวลาแค่สองสัปดาห์

ปัญหาต่อมาคือ จะรู้ได้อย่างไรว่าโค้ดถูกเปลี่ยนจนกระทบการทำงานจริงๆ ไม่ใช่แค่การเคาะบรรทัด หรือใส่คอมเมนต์เพิ่ม? วิธีแก้ที่นำมาใช้คือการทำ Structural Hash ระบบจะทำการตัดคอมเมนต์และช่องว่างทิ้งทั้งหมด (Normalize) ก่อนจะนำโค้ดไป Hash ซึ่งเขาเลือกใช้ `xxhash-rust` (`xxHash64`) เพราะทำงานได้เร็วระดับ Memory bandwidth ทำให้อัลกอริทึมนี้ไม่เคยเป็นคอขวดของระบบ และไม่ได้ต้องการความปลอดภัยระดับ Cryptography อย่าง SHA-256 แต่จุดที่ต้องระวังคือ แม้จะตัดช่องว่างทิ้ง แต่ระบบจะไม่ไปยุ่งกับชื่อตัวแปรหรือชื่อฟังก์ชัน เพราะการเปลี่ยนชื่อ Public function ย่อมส่งผลกระทบต่อการทำงานแน่นอน

แต่โค้ดจริงมีการย้ายที่และเปลี่ยนชื่อตลอดเวลา การจะจับคู่ (Match) Entity ระหว่างโค้ดเวอร์ชันเก่ากับเวอร์ชันใหม่จึงเป็นเรื่องยาก ระบบนี้แก้ปัญหาด้วยอัลกอริทึมจับคู่ 3 สเต็ป เริ่มจาก Phase 1 เทียบด้วย Entity ID ตรงๆ (O(n) ผ่าน Hash map) ซึ่งจะจับคู่พวกที่ไม่ได้เปลี่ยนชื่อหรือย้ายที่ได้เกือบหมด สิ่งที่เหลือรอดจะถูกส่งไป Phase 2 เทียบด้วย Structural Hashes (O(n)) เพื่อจับคู่โค้ดที่ถูกเปลี่ยนชื่อหรือย้ายตำแหน่งแต่ไส้ในเหมือนเดิม และสุดท้ายสำหรับพวกที่เปลี่ยนจนจำแทบไม่ได้ จะตกไปสู่ Phase 3 การใช้ Jaccard Similarity เพื่อเทียบความคล้ายคลึงของชุด Token แม้การคำนวณขั้นนี้จะเป็น O(n²) แต่เพราะ Phase ก่อนหน้าได้คัดกรอง Entity ไปจนแทบไม่เหลือแล้ว การคำนวณจริงจึงเกิดกับ Entity แค่หยิบมือ ทำให้ระบบตอบสนองได้แบบแทบจะในทันที แม้ไฟล์ Diff จะใหญ่แค่ไหนก็ตาม

ความเจ๋งอีกอย่างของการมองโค้ดเป็น Entity คือการสร้าง Dependency Graph ผ่าน `petgraph` ระบบจะสร้างกราฟทิศทาง (Directed Graph) เพื่อหาว่าคลาสไหนเรียกใช้ฟังก์ชันไหน ทำให้เราสามารถตอบคำถามสุดคลาสสิกได้ว่า ถ้าระเบิดฟังก์ชันนี้ รัศมีผลกระทบจะไปถึงไหนบ้าง โดยระบบจะใช้การค้นหาแบบ BFS บนกราฟที่ถูกกลับทิศ เริ่มต้นจาก Entity ที่ถูกแก้ไข การเลือกใช้ BFS แทน DFS ไม่ใช่เรื่องบังเอิญ แต่มันช่วยจัดลำดับผลลัพธ์ตามระยะห่าง ผู้เรียกใช้ฟังก์ชันเราโดยตรงจะโผล่ขึ้นมาก่อน ทำให้เรารู้ได้ทันทีว่าความเปลี่ยนแปลงนี้จะกระทบใครบ้าง ข้อมูลระดับนี้เป็นสิ่งที่มองไม่เห็นเลยใน Git diff แบบธรรมดา

ถ้าถามว่าสเกลงานทั้งหมดนี้ทำไมถึงรันได้ลื่นไหล คำตอบคือ `Rayon` และ Borrow Checker ของ #Rust ครับ การ Parse ไฟล์ในโปรเจกต์เป็นงานที่แยกอิสระจากกัน ภาษา Rust อนุญาตให้เราเปลี่ยนโค้ดจากการทำงานแบบเส้นตรงเป็นการประมวลผลแบบขนานได้ด้วยการแทรกคำสั่ง `.par_iter()` แค่คำเดียว โดยที่เราไม่ต้องมานั่งปวดหัวเรื่อง Thread safety หรือ Data race เลย เพราะ Borrow Checker การันตีให้ตั้งแต่ตอนคอมไพล์แล้วว่าไม่มี Closure ไหนเผลอไปจับ Mutable reference ของ Shared state ผลลัพธ์คือการอ่านไฟล์หลายพันไฟล์เสร็จภายในหลักวินาที ผ่านการทำงานของ Work-stealing scheduler ที่บาลานซ์โหลดให้ฟรีๆ โดยไม่ต้องพึ่งพา Mutex หรือ Atomic เลยสักนิด

จากทั้งหมดที่เล่ามา เครื่องมือนี้ไม่ได้มีไว้แค่ดู Diff แต่ยังมาพร้อมกับ `weave` ซึ่งทำหน้าที่เป็น Custom Merge Driver ที่สามารถเสียบเข้ากับ `.gitattributes` ได้เลย เวลามีการ Merge โค้ด มันจะใช้ประโยชน์จาก `parent_id` ในการทำ Inner merge ยกตัวอย่างเช่น ถ้ามีนักพัฒนาสองคนแก้โค้ดคนละ Method แต่บังเอิญอยู่ใน Class เดียวกัน Git แบบเดิมจะฟาด Conflict ทันทีเพราะมองเป็นบล็อกข้อความเดียวกัน แต่ weave จะแตก Class นั้นออกเป็น Method ย่อยๆ และทำ Merge แยกกัน ทำให้ผ่านไปได้อย่างสะอาดหมดจด และเพื่อความปลอดภัยสูงสุด มันถูกออกแบบมาแบบ Conservative คือถ้าระบบมันไปต่อไม่ได้จริงๆ มันจะยอมหลีกทางและโยนงานกลับไปให้ระบบ Line-level merge ของ Git ตามปกติ ทำให้เราสามารถเปิดใช้งานระบบนี้ทิ้งไว้ได้เลยโดยไม่ต้องกลัวพัง

ใครที่สนใจสถาปัตยกรรมสวยๆ หรืออยากเห็นการใช้งาน `tree-sitter` ร่วมกับ Graph algorithm และ Concurrency หนักๆ ในโลกของ #Rust สามารถไปลองโหลดมาเล่นกันได้ รับรองว่าเปิดโลกแน่นอน
