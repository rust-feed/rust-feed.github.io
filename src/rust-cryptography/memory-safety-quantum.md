# เมื่อ Memory Safety คือด่านแรกที่คณิตศาสตร์ควอนตัมเจาะไม่ได้

> 📅 วันที่เผยแพร่: 2026-04-23

ชวนคุยเรื่องควอนตัมกันบ้างนะครับ เนื้อหาอ้างอิงมาจากบทความของ น้องปูส้มโจรสลัด จากงานวิจัยล่าสุดที่ชี้ว่าขอบเขตควอนตัมอาจใกล้กว่าที่คาด หรือแม้แต่ Cloudflare และ Meta ที่ประกาศเป้าหมายจะอัปเกรดระบบเป็น Post-Quantum เต็มรูปแบบภายในปี 2029 แต่ในมุมมองของวิศวกรระบบความปลอดภัยนั้น เรามักจะพูดกันเสมอว่า วันที่คุณอ่านข่าวการโจมตีด้วยควอนตัมในสื่อหลัก มันสายเกินไปแล้ว เพราะภัยคุกคามในรูปแบบ Harvest Now, Decrypt Later กำลังเกิดขึ้นเงียบๆ มานาน คำนี้เริ่มพูดถึงกันบ่อยขึ้น

แต่สิ่งที่น่ากังวลกว่าคณิตศาสตร์ควอนตัม คือช่องโหว่ในการเขียนโค้ดจริงๆ จากงานวิจัยเชิงประจักษ์ของ MIT พบว่าในไลบรารีเข้ารหัสกว่า 37.2% ของช่องโหว่ทั้งหมด เกิดจากปัญหา Memory Safety ในขณะที่มีเพียง 27.2% เท่านั้นที่มาจากความผิดพลาดทางคณิตศาสตร์หรืออัลกอริทึม การโจมตีใน Production ส่วนใหญ่ไม่ได้เจาะผ่าน Shor’s Algorithm โดยตรง แต่เจาะผ่าน Buffer Overflow การจัดการ Lifecycle ของคีย์ที่ผิดพลาด หรือการประกอบส่วนประกอบเข้ารหัสผิดวิธี ดังนั้นนี่จึงเป็นจุดที่ Rust แสดงศักยภาพชัดเจนที่สุด

ระบบ Ownership และ Borrow Checker ที่บังคับให้จัดการหน่วยความจำและวงจรชีวิตของข้อมูลสำคัญอย่างคีย์หรือ Nonce ตั้งแต่ขั้นตอน Compile Time ซึ่งตัดปัญหา Use After Free หรือการนำคีย์มาใช้ซ้ำที่นำไปสู่หายนะทางความปลอดภัยได้เกือบทั้งหมด ในเชิงสถาปัตยกรรม เรามีทางเลือกหลักสองทางสำหรับระบบปัจจุบัน ฝั่งแรกคือ `aws-lc-rs` ที่เหมาะมากกับงาน Server-Side เพราะทำงานเร็วและอยู่ภายใต้การดูแลของ AWS แต่ข้อจำกัดคือมันเป็น Wrapper ของไลบรารีภาษา C ทำให้ไม่สามารถทำงานบนทุก Platform ได้ เช่น WebAssembly ฝั่งที่สองคือ Ecosystem ของ Pure Rust Crypto ซึ่งหลายแพ็กเกจผ่านการ Audit จากบุคคลที่สามอย่างเป็นทางการ ทำงานได้ Cross-Platform โดยไม่ต้องพึ่ง Native Wrapper ช่วยลดจุดโจมตีจาก FFI ไปได้อย่างสิ้นเชิง การเลือกใช้อาจขึ้นอยู่กับ Architecture ของคุณ แต่แนวโน้มของวงการกำลังมุ่งไปสู่ Pure Rust 😎 เพื่อความยืดหยุ่นและความปลอดภัยระดับลึก

การเลือก Hash Function ที่เหมาะสมคือด่านแรก SHA-512 ยังเป็นตัวเลือกที่สมดุลที่สุดระหว่างความเร็วและความปลอดภัย โดยให้ความเร็วเกิน 1GB/s ซึ่งเพียงพอสำหรับงานทั่วไป และควรใช้ SHA-256 เฉพาะกรณีที่จำเป็นต้องทำ Deduplication ที่ต้องการความเร็วสูงสุดจริงๆ แต่สำหรับข้อมูลความน่าเชื่อถือต่ำอย่างรหัสผ่านหรือ One-Time Code เราต้องใช้ Argon2id (เวอร์ชัน v19) เท่านั้น เนื่องจากออกแบบมาให้เป็น Memory-Hard Algorithm ที่ต้านทานการโจมตีด้วย GPU หรือ ASIC ได้โดยตรง

ส่วนกระบวนการ Key Derivation ที่ใช้แยก Master Key ออกเป็น Subkey สำหรับงานต่างๆ หรือทำ Ratcheting ในโปรโตคอลเข้ารหัส ควรใช้ SHAKE256 (ตามมาตรฐาน FIPS 202) เป็นหลัก เพราะรองรับการผูก Context และ Length ของข้อมูลได้ชัดเจน ป้องกัน Collision ในกรณีที่คีย์ถูกนำไปใช้ซ้ำหลายวัตถุประสงค์ หากสภาพแวดล้อมไม่รองรับ HKDF-SHA512 ก็ยังถือเป็นตัวเลือกสำรองที่ปลอดภัยและใช้งานง่าย

ในส่วนของ Symmetric Encryption หรือ AEAD วงการเริ่มเห็นตรงกันแล้วว่า AES-256-GCM แม้จะเป็นมาตรฐานระดับ Military-Grade และปลอดภัยต่อ Compliance แต่มีจุดอ่อนเชิงโครงสร้างเรื่อง Nonce ที่สั้นเกินไป และการใช้งานซ้ำเพียงครั้งเดียวก็ทำให้ระบบพังทลายได้ รวมถึงขาดกลไกการผูกบริบทข้อมูล ที่อาจนำไปสู่ช่องโหว่ในโลกจริงหาก Metadata ถูกดัดแปลง XChaCha20-Poly1305 แก้ปัญหาเรื่อง Nonce ยาวขึ้นแต่ยังขาดกลไกการผูกบริบทข้อมูลเช่นกัน

ดังนั้นหากไม่ติดข้อบังคับทางกฎหมาย ChaCha20-BLAKE3 จึงเป็นคำแนะนำหลัก เพราะออกแบบมาผูกบริบทการเข้ารหัสทั้งหมดไว้ด้วยกันตั้งแต่แรก ทำให้ปลอดภัยต่อการโจมตีแบบสลับบริบทอย่างมีนัยสำคัญ สำหรับการเก็บ Secrets บน Backend ควรใช้สถาปัตยกรรม Envelope Encryption คู่กับ KMS ที่รองรับ Hardware-Backed API เพื่อให้ Data Key ถูกเข้ารหัสแยกต่างหาก และบริการ KMS ไม่จำเป็นต้องเห็นข้อมูลจริงเลย

ส่วน Asymmetric Cryptography ความเห็นพ้องต้องกันในวงการคือเราต้องใช้ Hybrid Key Exchange ที่รวม Classical Elliptic Curve (เช่น X25519) เข้ากับ Post-Quantum KEM ปัจจุบัน X-Wing (ซึ่งผสาน X25519 กับ ML-KEM-768) กำลังเป็น Draft มาตรฐานของ IETF ที่ได้รับความนิยมนำไปใช้งานจริง ส่วนฝั่ง Asymmetric Signatures วงการยังไม่ได้เห็นพ้องเรื่อง Hybrid Signature มากนัก ดังนั้นคำแนะนำในปัจจุบันคือใช้ Pure Post-Quantum ไปเลย ด้วย ML-DSA-65 หรือ ML-DSA-87 (FIPS 204)

หากต้องการความปลอดภัยสูงสุด สำหรับ JWT ให้ใช้ HMAC-SHA512 ในระบบ Monolithic หรือ ML-DSA-65 หากเป็นระบบ Asymmetric ส่วนโปรโตคอล End-to-End Encryption นั้น RFC 9180 (HPKE) คือมาตรฐานพื้นฐานที่ควรยึดถือ โดยจับคู่ KEM: X-Wing, AEAD: AES-256-GCM, และ KDF: HKDF-SHA512 เพื่อให้ได้ความสมดุลระหว่างความปลอดภัยและการทำงานร่วมกันได้

ในระดับ Transport Layer ปัจจุบัน TLS รองรับเฉพาะ X25519MLKEM768 เป็นอัลกอริทึมแลกเปลี่ยนคีย์ที่ต้านทานควอนตัมได้ ต้องตรวจสอบให้แน่ใจว่า Load Balancer หรือ Reverse Proxy ของรองรับ Cipher Suite นี้แล้ว (ไลบรารีอย่าง aws-lc-rs หรือ Pingoo รองรับอยู่แล้ว) แต่สิ่งที่สำคัญกว่าการเปลี่ยนอัลกอริทึมคือ Cryptographic Agility องค์กรซอฟต์แวร์ควรสร้างกระบวนการที่พร้อมสลับ Primitive ได้ทันทีเมื่อพบจุดอ่อน หรือมีการอัปเดตมาตรฐาน โดยควรวาง Roadmap Migration ให้เสร็จก่อนสิ้นปี 2026 เพื่อรองรับ Compliance ที่จะเปลี่ยนเกณฑ์เป็น Mandatory ในอนาคต การย้ายไปใช้ Post-Quantum ไม่ใช่จุดสิ้นสุด แต่คือการพิสูจน์ว่าสถาปัตยกรรมของเราพร้อมรับมือกับการเปลี่ยนแปลงของคณิตศาสตร์และความปลอดภัยในยุคใหม่ครับ

การเข้าใจแก่นของ Type System, Zero-Cost Abstractions และ Memory Safety Model คือรากฐานที่ช่วยให้เราประกอบส่วนประกอบเข้ารหัสได้อย่างถูกต้องตั้งแต่ขั้นตอน Compile Time เมื่อพายุควอนตัมกำลังก่อตัว การวางรากฐานระบบบน Pure Rust Ecosystem ที่ผ่านการ Audit ผสมผสานกับสถาปัตยกรรมที่เน้นการผูกบริบทข้อมูลและ Hybrid Design คือทางรอดที่ควบคุมได้จริง
