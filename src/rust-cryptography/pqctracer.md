# เรื่องราวของ "pqctracer" library เล็กๆ ที่แก้ปัญหาใหญ่ที่หลายคนมองข้าม

> 📅 วันที่เผยแพร่: 2026-02-28

สมมติว่าวันนี้คุณเขียน Rust app ที่ยิง HTTPS request ไปหา API ต่างๆ อยู่ทุกวัน คุณใช้ reqwest + rustls ซึ่งเป็น combination ที่ดีมาก และคุณก็รู้ว่า rustls รองรับ Post-Quantum Cryptography (PQC) แล้วด้วย ฟังแล้วก็ดูปลอดภัยดีว่าไหมครับ แต่มีคำถามหนึ่งที่ตอบได้ยากกว่าที่คิด connection ที่แอปคุณทำอยู่จริงๆ ตอนนี้ มันใช้ Post-Quantum จริงหรือเปล่า ถ้าตอบว่า "ก็น่าจะใช่ เพราะ rustls รองรับแล้ว" ถ้าตอบได้แค่นี้ผมว่าเรามาดู pqctracer กันดีกว่าครับว่าจะให้คำตอบอะไรแก่เราเพิ่มเติมบ้างไหม

ก่อนอื่น ทำความเข้าใจก่อนครับว่า HTTPS ทำงานยังไง ทุกครั้งที่แอปเราเปิด connection ไปยัง https:// อะไรก็ตาม มันจะมีขั้นตอนที่เรียกว่า TLS Handshake เกิดขึ้นก่อนเสมอ คิดว่ามันเหมือนการที่สองฝ่ายนัดหมายกันก่อนคุยว่า "เราจะเข้ารหัสข้อมูลกันด้วยวิธีไหนดี" ส่วนหนึ่งของการนัดหมายนี้คือการตกลงกันเรื่อง Key Exchange หรือ "วิธีสร้างกุญแจลับร่วมกัน" โดยที่ไม่ต้องส่งกุญแจจริงๆ ผ่านเน็ต (เพราะถ้าส่งกุญแจผ่านเน็ตก็โดนดักได้) วิธีที่นิยมใช้ทุกวันนี้ชื่อว่า X25519 มันแก้ไม่ได้สำหรับคอมพิวเตอร์ทั่วไป แต่ Quantum Computer สามารถแก้ได้ด้วยสิ่งที่เรียกว่า Shor's Algorithm

NIST (องค์กรมาตรฐานของสหรัฐฯ) จึงได้ประกาศมาตรฐานการเข้ารหัสแบบใหม่ที่ทนต่อ Quantum Computer เรียกว่า Post-Quantum Cryptography (PQC) โดย algorithm หลักที่เลือกมาสำหรับ Key Exchange คือ ML-KEM (ชื่อเดิมคือ CRYSTALS-Kyber) ซึ่งอยู่บนโจทย์คณิตศาสตร์ที่ยังไม่มีคอมพิวเตอร์ชนิดไหนแก้ได้อย่างมีประสิทธิภาพ ไม่ว่าจะ classical หรือ quantum แต่การเปลี่ยนทันทีทั้งหมดทำได้ยาก เพราะ server หลายแสนเครื่องทั่วโลกยังใช้ algorithm แบบเดิม ทางออกในทางปฏิบัติจึงเป็นการใช้ Hybrid Scheme รวม X25519 แบบเก่าเข้ากับ ML-KEM แบบใหม่ ผลลัพธ์คือ algorithm ที่ชื่อว่า X25519MLKEM768 ซึ่งปลอดภัยทั้งจากคอมพิวเตอร์ธรรมดาและ Quantum Computer ถ้า algorithm ใดอันหนึ่งถูก break ในอนาคต อีกอันก็ยังปกป้องอยู่

Chrome รองรับ X25519MLKEM768 มาตั้งแต่เวอร์ชัน 124 แล้ว rustls ก็รองรับเช่นกัน แต่นั่นก็แปลว่า ทุก connection จะปลอดภัย ไหมคำตอบคือ ไม่เสมอไปครับ เพราะ Key Exchange ต้องมีทั้งสองฝ่ายเห็นตรงกัน เวลา client ส่ง TLS Handshake ออกไป มันจะบอก server ว่า "ฉันรองรับ Key Exchange แบบเหล่านี้นะ" แล้วเรียงตามลำดับที่ชอบ server ก็จะ เลือกตัวแรกในลิสต์ที่ตัวเองรองรับ เสมอ ถ้า server รองรับ X25519MLKEM768 → connection ได้ PQC ถ้า server ยังไม่รองรับ → เลือก X25519 แบบเดิมแทน และที่สำคัญที่สุด มันเกิดขึ้น โดยไม่มีข้อผิดพลาดใดๆ ทั้งสิ้น response ยังเป็น 200 OK app ยังทำงานปกติ คุณไม่มีทางรู้เลย

ถ้าแอปของคุณเรียก API สิบ-ยี่สิบเจ้า คุณรู้ไหมว่าเจ้าไหนรองรับ PQC แล้ว? ถ้าไม่รู้ pqctracer ช่วยได้ครับ 5555 เล่ามาตั้งนานกว่าจะเข้าเรื่อง

pqctracer คือ Rust library ที่ให้ HTTP client แบบ "ฉลาดขึ้นนิดหน่อย" มันทำทุกอย่างที่ reqwest ทำได้ แต่เพิ่มความสามารถในการ รายงานว่า TLS Handshake ที่เพิ่งเกิดขึ้นใช้ Key Exchange Group อะไร มันไม่ได้ประดิษฐ์อะไรใหม่ แค่ห่อ reqwest และ rustls เข้าด้วยกัน แล้วดึงข้อมูลที่ rustls รู้อยู่แล้วออกมาให้คุณเห็น ผลที่ได้คือ result.group จะบอกว่าใช้ X25519MLKEM768 (ปลอดภัย) หรือแค่ X25519 (แบบเดิม) และ result.cipher บอก cipher ที่ใช้ เช่น TLS13_AES_256_GCM_SHA384 ข้อมูลนี้มาจาก TLS Handshake จริงๆ ไม่ใช่การเดา

ถ้าจะจำเรื่องนี้แค่ประโยคเดียว "ใช้ HTTPS" กับ "TLS Handshake ใช้ Post-Quantum Crypto" คือสองเรื่องที่ต่างกันโดยสิ้นเชิง และคุณจะไม่รู้เลยว่าเรื่องที่สองเกิดขึ้นหรือเปล่า ถ้าไม่มีเครื่องมือดูลงไปที่ Handshake จริงๆ pqctracer เป็น library เล็กๆ ที่แก้ปัญหาจุดนี้ได้ตรงๆ และ API ก็ไม่ซับซ้อนเลย ลองเอาไป audit dependencies ของ project ดูสักครั้งก็ได้ครับ ผลลัพธ์อาจน่าแปลกใจกว่าที่คิด
