# มุมมองจากคนเขียน Go สิบปี ที่ลองจับรัสต์ครั้งแรกอย่างจริงจัง

> 📅 วันที่เผยแพร่: 2026-04-27

"ผม" ในบทความนี้หมายถึง Paul Hinze [software engineer]

ผมเขียนภาษา Go มาเกือบสิบปีแล้ว ตลอดเวลานั้นภาษารัสต์คือภาษาที่ผมเคารพแต่ยืนมองอยู่ห่างๆ เคยซื้อหนังสือมาลองพลิกอ่าน เคยลองเล่น rustlings สองสามรอบ แต่สุดท้ายก็วนกลับมาที่ Go เหมือนเดิม เพราะขาด "เหตุผลที่ใช่" ที่จะผลักดันให้ลงมือเรียนรู้อย่างจริงจัง จนกระทั่งมีโอกาสได้ไปร่วมงาน TokioConf และอยากทำเดโมอะไรสักอย่างให้ทันงาน เลยตัดสินใจว่านี่แหละคือจังหวะที่เหมาะสมที่สุดที่จะลองจับรัสต์แบบจริงจัง

โปรเจกต์ที่เลือกคือเซิร์ฟเวอร์แชทง่ายๆ ที่ทำงานได้จริง แทนที่จะนั่งงมเอกสารคนเดียว ผมใช้ Claude เป็นเหมือนคู่หูโปรแกรมเมอร์รัสต์ที่คอยอธิบายโค้ดให้ฟังทีละบรรทัด แม้จะไม่ใช่การนั่งคู่กับมนุษย์จริงๆ แต่อย่างน้อยมันก็อดทนและพร้อมตอบคำถามแม้ตอนนั่งบนเครื่องบินที่ไม่มีอินเทอร์เน็ต ผลลัพธ์คือแอปแชทที่รันอยู่บน <https://chat.miren.toys> ซึ่งใครที่สนใจก็สามารถเข้าไปลองพิมพ์เล่นกันได้

พอได้ลงมือเขียนจริงๆ จังๆ สมองที่คุ้นเคยกับรูปแบบของ Go ก็ต้องปรับตัวหลายอย่าง สิ่งแรกที่รู้สึกเลยว่า "นี่แหละสิ่งที่ Go ควรมี" คือระบบ Enum ที่ตรวจสอบความครบถ้วนโดยคอมไพเลอร์ ใน Go เวลาเราเพิ่มกรณีใหม่ให้กับประเภทข้อมูล เรามักต้องเขียนเทสหรือใช้โค้ดเจนเพื่อเช็คว่าเราจัดการทุกกรณีครบหรือยัง แต่ในรัสต์ ถ้าเราลืมจัดการกรณีใดกรณีหนึ่ง คอมไพเลอร์จะฟ้องทันทีตั้งแต่ขั้นตอนคอมไพล์ ไม่ต้องรอให้บั๊กโผล่ในโปรดักชัน

อีกจุดที่ทำให้รู้สึกแปลกใหม่คือตัวดำเนินการ `?` ที่ใช้จัดการข้อผิดพลาด ใน Go เราชินกับการเขียน `if err != nil` ซ้ำๆ จนจำได้อัตโนมัติ แต่ในรัสต์ แค่ใส่ `?` ตัวเดียวก็จัดการทั้งการเช็คและรีเทิร์นข้อผิดพลาดได้ทันที แม้ช่วงแรกจะรู้สึกไม่ค่อยสบายใจกับการที่ฟังก์ชันมีจุดออกหลายทางแบบอ้อมๆ แต่พอนานเข้าก็เริ่มเข้าใจว่านี่คือการที่ภาษาวางใจให้ระบบไทป์เป็นผู้รับผิดชอบ แทนที่จะบังคับให้เราเขียนทุกขั้นตอนอย่างชัดเจนแบบ Go

มีบรรทัดหนึ่งที่ทำให้ผมขำออกมาดังๆ ตอนอ่านโค้ดครั้งแรก คือการเขียน pattern match ซ้อนกันหลายชั้นเพื่อเช็กว่าข้อมูลที่ได้มาถูกต้องครบทุกสเต็ป อ่านแล้วเหมือนมีคนกำลังพึมพำกับตัวเองว่า "โอเค... มีข้อมูล... โอเค... เป็นข้อความ... เอาล่ะใช้ได้" แม้ตอนแรกจะรู้สึกว่าโค้ดแบบนี้ดูซับซ้อนเกินไป แต่พอลองรีแฟกเตอร์มาใช้ `?` แทน ก็พบว่ามันทำงานแทน if ซ้อนกันห้าชั้นใน Go ได้แบบไม่พลาดสักชั้น แม้การอ่านอาจจะยังรู้สึกแปลกๆ แต่ความมั่นใจที่ได้มานั้นคุ้มค่า

ส่วนที่ท้าทายที่สุดสำหรับคนมาจากโลก Go คือเรื่อง Generic และ Trait Constraint ในรัสต์ บางครั้งเราเจอโค้ดที่เขียน Constraint ยาวเป็นแถว จนต้องหยุดอ่านหลายรอบกว่าจะเข้าใจว่าฟังก์ชันนี้รับอะไรได้บ้าง และต้อง implement อะไรบ้าง ใน Go เองก็มีจุดที่ซับซ้อนแบบนี้เหมือนกัน แต่รัสต์ดูเหมือนจะพาเราไปถึงจุดนั้นเร็วกว่านิดหน่อย สำหรับแอปทั่วไป เราอาจเลือกเขียนด้วย concrete type ไปเลยก็ได้ แต่ถ้าอยากได้ความยืดหยุ่นจริงๆ การลงทุนเรียนรู้ระบบไทป์ของรัสต์ก็ให้ผลตอบแทนที่คุ้มค่า

สิ่งที่ทำให้ผมต้องเปลี่ยนมุมมองอย่างสิ้นเชิงคือการจัดการ Async ในรัสต์ ใน Go เราแค่พิมพ์ `go func()` แล้วระบบรันไทม์ก็จัดการทุกอย่างให้เบื้องหลัง ทั้ง Goroutines การจัดตารางงาน GC ทุกอย่างซ่อนอยู่ใต้พรมที่เราเปิดดูไม่ได้ แต่รัสต์ดึงพรมผืนนั้นออกหมด ภาษามอบแค่ไวยากรณ์สำหรับ async ไว้ให้ ส่วนรันไทม์จริงๆ คือไลบรารีที่เราต้องเลือกและนำเข้าเอง เช่น Tokio การที่เราต้องประกาศใช้รันไทม์อย่างชัดเจน ทำให้เราเห็นโครงสร้างของระบบมากขึ้น รู้ว่าแต่ละทาสก์ทำงานอย่างไร ช่องทางสื่อสารเชื่อมต่อกันแบบไหน ไม่ใช่กล่องดำที่ทำงานให้โดยเราไม่รู้ว่าข้างในเป็นยังไง

ตอนแรกผมคิดว่าความ "เยอะ" ของโค้ดรัสต์คือข้อเสีย แต่พอนานเข้าก็เริ่มเห็นว่าความเยอะนั้นคือความชัดเจน ในทางกลับกัน Go อาจจะดูเขียนง่าย แต่ซ่อนความซับซ้อนไว้ข้างในเยอะกว่าที่เราคิด รัสต์บังคับให้เราต้องคิดและเขียนทุกอย่างออกมาอย่างตั้งใจ ไม่มีอะไรที่ถูกซ่อนไว้

เครื่องมืออย่าง tokio-console ก็ช่วยให้เห็นภาพนี้ชัดเจนยิ่งขึ้น ถ้าใน Go เรามี pprof ที่ช่วยสแนปช็อตประสิทธิภาพของระบบ tokio-console ก็ทำหน้าที่คล้ายกันแต่แสดงข้อมูลแบบเรียลไทม์ เราเห็นทุกทาสก์ ทุกช่องทาง ทุกมิวเท็กซ์ ทำงานอยู่ตรงไหน ใช้เวลากี่ไมโครวินาที การเชื่อมต่อเพียงสามบรรทัดกับไฟล์คอนฟิกเล็กน้อย ก็ทำให้เราดีบัคระบบแอสซินโครนัสได้อย่างแม่นยำ แม้แต่โค้ดที่เราเขียนเองก็ปรากฏในคอนโซลนี้ได้อย่างเป็นธรรมชาติ ราวกับเป็นส่วนหนึ่งของรันไทม์เอง

หลังจากใช้เวลาหลายชั่วโมงกับรัสต์อย่างจริงจัง ผมต้องยอมรับว่าสนุกกับการเรียนรู้มาก แต่ก็ไม่ได้หมายความว่าผมจะเปลี่ยนจาก Go มาเป็นปูเต็มตัวในทันที Go ที่ผมทำไว้ยังตอบโจทย์งานส่วนใหญ่ที่ผมทำอยู่ทุกวัน แต่ผมก็เริ่มเข้าใจแล้วว่าทำไมหลายคนถึงบอกว่า "นานพอ คุณอาจจะเปลี่ยนใจ" รัสต์ไม่ใช่ภาษาที่ง่ายกว่า แต่เป็นภาษาที่บังคับให้เราคิดอย่างมีระบบมากขึ้น ตั้งแต่การจัดการหน่วยความจำ ไปจนถึงการออกแบบไทป์และคอนเคอร์เรนซี

สำหรับทีมพัฒนาที่อาจกำลังลังเลระหว่างสองภาษานี้ คำตอบอาจไม่ใช่การเลือกข้าง แต่เป็นการเข้าใจว่าแต่ละภาษาออกแบบมาเพื่อแก้ปัญหาแบบไหน Go เหมาะกับการสร้างระบบที่เร็ว ง่าย และดูแลได้โดยทีมหลากหลายระดับ รัสต์เหมาะกับการสร้างระบบที่ต้องการความมั่นใจระดับสูง ปลอดภัยจากข้อผิดพลาดเชิงหน่วยความจำ และพร้อมทำงานในสภาพแวดล้อมที่ซับซ้อน การได้ลองสัมผัสทั้งสองโลกทำให้เห็นว่าไม่มีภาษาไหน ดีกว่าในทุกสถานการณ์ มีแต่ภาษาที่เหมาะกับงานในขณะนั้นมากกว่า

และบางที การเป็นโปรแกรมเมอร์ที่ดีในยุคนี้ อาจไม่ใช่การยึดติดกับภาษาใดภาษาหนึ่ง แต่คือการรู้จักเลือกเครื่องมือให้เหมาะกับปัญหา และพร้อมเรียนรู้สิ่งใหม่เมื่อโจทย์เปลี่ยนไป ไม่ว่าคุณจะเป็นโกเฟอร์หรือปู สิ่งสำคัญที่สุดคือโค้ดที่เขียนออกมาต้องตอบโจทย์ผู้ใช้ และดูแลได้ต่อไปในอนาคต
